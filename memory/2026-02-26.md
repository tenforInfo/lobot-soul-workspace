## 📋 讨论概要

## 🎯 核心议题

### 1. 跑力值(RQ)定义与计算

**问题:**
- 当前市面上的RQ展示方式不够直观（上月vs本月曲线对比繁琐）
- 平均心率+平均配速的比较方式有问题（心率漂移导致难以评估）

**方案:**
- 需要**自定义跑力值算法**（暂时叫"阿Q"）
- 参考RQ跑力值作为基础，但要比它更精确
- 核心目标：提炼一个指标来评价当前能力是提高还是降低

---

### 2. 数据清洗与分析

**问题:**
- 直接取整段的平均心率和配速不准确
- 存在心率漂移、爬坡路段、配速波动等干扰因素

**解决方案:**
1. **去除异常数据:**
   - 去掉爬坡路段（有坡的数据不要）
   - 去掉前1公里和最后部分的配速（不稳定）
   - 去掉极高/极低的心率异常值

2. **提取有效区间:**
   - 在15秒配速区间内，跑了半分钟以上的视为有效
   - 只保留平路数据（无坡）
   - 按照有氧配速评估当前能力

3. **比较方式:**
   - 评估5公里平路有氧配速
   - 对比历史同期数据看进步/退步

## 🔧 补充讨论（续）

### 4. 数据获取与分析

**数据来源:**
- 需要登录获取历史数据
- 对比新一代数据和旧一代数据，看看缺什么字段

**数据字段精简:**
- 区分必要字段 vs 可选字段
- 针对"带心率带"和"不带心率带"的情况做区分（不带时缺少心率相关字段）
- 目标：节省存储/传输，只保留关键数据

### 5. 界面规划

需要设计**两个界面**：
1. **数据/分析界面** - 展示清洗后的数据、跑力值趋势
2. **计划界面** - 根据数据调整训练计划

**核心概念 - "作战仓":**
- 需要一个核心数据仓库/模块来统筹所有数据
- 作为数据流转和算法计算的中心枢纽

### 6. 迭代思路

```
获取数据 → 分析差异 → 精简字段 → 设计界面 → 调整计划
```

**关键原则：**
- 先根据现有数据设计算法
- 再根据数据反推需要什么字段
- 最后根据计划需求调整数据收集
